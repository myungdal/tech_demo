## 📋 .cursorrules 활용

### 프로젝트 규칙 자동 적용

```
nearest3/
├── .cursorrules                    # 전체 공통 규칙
├── server/.cursorrules             # 서버 전용 규칙
└── client/Client/Source/.cursorrules  # 클라이언트 전용 규칙
```

### 규칙 예시

```markdown
## 파일 인코딩
- UTF-8 with BOM
- CRLF 줄바꿈

## 네이밍 규칙
- 클래스: PascalCase
- 멤버 변수: m + PascalCase
- 전역 변수: g + PascalCase

## 자동 생성 파일
- autoGenerated 폴더/파일은 직접 수정 금지
```

## 💬 효과적인 AI 프롬프트

### 1. 기능 추가 요청

```
❌ 나쁜 예:
"우편 기능 만들어줘"

✅ 좋은 예:
"새 우편이 도착하면 클라이언트에 푸시 알림을 보내는 기능을 추가해줘.
- NOTIFY_NEW_MAIL 패킷 사용
- 수신자가 온라인일 때만 전송
- 기존 MailSystem 클래스에 추가"
```

### 2. 버그 수정 요청

```
❌ 나쁜 예:
"이거 왜 안돼?"

✅ 좋은 예:
"MainServerApp::OnPacketReceived에서 nullptr 크래시 발생.
- 재현: 클라이언트 비정상 종료 후 패킷 수신 시
- 로그: [ERROR] Access violation at 0x00000000
- 예상 원인: 세션 종료 후 패킷 처리 타이밍 이슈"
```

### 3. 리팩토링 요청

```
❌ 나쁜 예:
"이 코드 좀 정리해줘"

✅ 좋은 예:
"UserManager 클래스 리팩토링 요청:
- 현재 문제: 2000줄 단일 파일, SRP 위반
- 목표: UserCache, UserLoader, UserValidator로 분리
- 제약: 기존 public API는 유지"
```

## 🎯 AI 활용 베스트 프랙티스

### DO ✅

```
• 명확하고 구체적인 요구사항 전달
• 컨텍스트(기존 코드, 제약사항) 제공
• AI 결과물 반드시 리뷰
• 점진적 개선 (한 번에 완벽 기대 X)
• 학습 목적으로도 활용 (왜 이렇게 했는지 질문)
```

### DON'T ❌

```
• 보안 정보 (API 키, 비밀번호) 전달
• AI 코드 무검증 사용
• 복잡한 비즈니스 로직 전체 위임
• 컨텍스트 없는 단순 명령
• AI 오류에 좌절 (반복 요청으로 개선)
```

## 🔧 MCP (Model Context Protocol) 활용

### 설정된 MCP 서버

```json
{
  "cursor-browser-extension": "웹 브라우저 제어",
  "cursor-ide-browser": "IDE 내 브라우저"
}
```

### 활용 예시

```
프론트엔드 개발 시:
1. 코드 수정
2. MCP로 브라우저 자동 새로고침
3. 결과 확인
4. AI가 스크린샷 분석
5. 추가 수정 제안
```
